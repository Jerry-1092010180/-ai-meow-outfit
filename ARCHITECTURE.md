## 1. 产品定位

**AI喵搭不是一套「拍照→重建→试衣」的单次工具。**

它是银泰百货 4500 万会员的 **永久数字身份系统**。

```
当前认知：   拍照 → 生成 GLB → 扔进 Three.js → 结束
正确认知：   注册 → 生成 Avatar → 持续更新 → 永久使用
                   ↕
          换衣、发型、妆容、配饰
                   ↕
          线下门店试穿、线上虚拟试穿、社交分享
```

**一句话**：Avatar 是会员的身体数字孪生，喵街 APP 是衣橱，银泰是商场。

---

## 2. 系统目标

| 目标 | 优先级 | 度量 |
|------|--------|------|
| ① 像本人 | P0 | 用户凭照片能认出是自己的 3D 模型 |
| ② 能长期保存 | P0 | 2 年后仍可访问，体形变化后可更新 |
| ③ 能换衣 | P0 | 同一 body 可穿戴不同服装 GLB |
| ④ 能分享 | P1 | 微信/朋友圈/小红书 一键分享 |
| ⑤ 永久身份 | P1 | Avatar ID 绑定喵街会员 ID，不可丢失 |
| ⑥ 单人可完成 | P2 | 无需专用设备，一部手机在家完成 |

---

## 3. 为什么当前方案走不到这里

当前是 **Visual Hull Silhouette Carving**。这是 1990 年代的算法，有根本性限制：

| 限制 | 原因 |
|------|------|
| **8 视图几何稀疏** | Visual Hull 需要 60+ 视图才能接近真实人体。8 视图 = "纸板人" |
| **轮廓丢失纹理** | GrabCut 扔掉所有颜色信息。结果永远没有皮肤、衣服细节 |
| **前置摄像头** | 自拍镜头画质差、手遮挡、姿态不可控。真正人体建模用后置相机 + 全身扫描 |
| **无参数化** | 每个重建是独立随机网格，无法与标准人体模板对应，无法换衣，无法动画 |
| **Mesh 拓扑不固定** | Marching Cubes 每次生成不同顶点顺序，不能复用服装资产 |
| **无持久化** | 重建完就结束，没有 Avatar ID，没有增量更新 |

**结论**：在当前基础上修修补补，永远无法达到"数字分身"的目标。需要更换底层数据表示。

## 4. 整体技术路线

### 核心决策：放弃离散网格 → 采用参数化人体模型（SMPL）

```
       旧方案                         新方案
  ┌─────────────────┐       ┌──────────────────────┐
  │ Visual Hull      │       │ SMPL-X                │
  │ Marching Cubes   │  →   │ (参数化人体模型)        │
  │ 随机拓扑网格      │       │ 人体形状参数 (β)       │
  │ 无纹理            │       │ 身体姿态参数 (θ)       │
  │ GLB = 最终资产    │       │ 纹理贴图 (独⽴)        │
  └─────────────────┘       └──────────────────────┘
```

**SMPL-X（Skinned Multi-Person Linear Model）** 是业界标准的人体参数化模型。它把人体表示为：
- **β (shape)**：身高、体重、体形等 10-300 个参数
- **θ (pose)**：全身 52 个关节的旋转角度
- **D (displacement)**：表面细节偏移（皱纹、肌肉线条）
- **Texture map**：独立于几何体的人体纹理贴图

**为什么选择 SMPL**：
- 拓扑固定（6045 顶点，无论高矮胖瘦都一样结构）
- 服装可叠加（因为拓扑固定，服装资产可标准化）
- 可动画（自带骨骼，可直接驱动行走/转圈）
- 可编辑（改 β 参数 = 改身材）
- 业界标准（MetaHuman、CLO3D、Blender 都支持）

## 5. 数据流（完整）

```
┌─────────────────────────────────────────────────────────────────────┐
│ A. 采集（一次性，可在门店或家中完成）                                │
├─────────────────────────────────────────────────────────────────────┤
│ 方案 1（推荐，质量最高）：门店 3D 体测仪                               │
│   → 60 秒全身扫描 → 点云 → SMPL 拟合 → 纹理贴图                      │
│ 方案 2（进阶级）：手机后置相机 + 全身视频                              │
│   → 用户穿着紧身衣，用后置相机缓慢旋转一周                               │
│   → 提取 30-60 帧 → 稀疏重建 + SMPL fitting                          │
│ 方案 3（入门级）：用户上传 2 张照片（正面 + 侧面）                      │
│   → HRNet/SMPLify 估计 SMPL 参数 → 从照片生成纹理                     │
│ 方案 4（当前最低门槛）：手机前置相机 8 张（保留作为兼容通道）            │
│   → 降级使用，只输出 β 参数，不生成纹理                                │
│                                                                     │
│ 无论哪种方案，输出都统一为：SMPL β 参数 + 纹理图 URL                    │
└─────────────────────────────────────────────────────────────────────┘
                                       ↓
┌─────────────────────────────────────────────────────────────────────┐
│ B. 后端管线（AIGC 4090D）                                           │
├─────────────────────────────────────────────────────────────────────┤
│ 输入：采集数据（点云 / 视频帧 / 照片 + 身高体重）                     │
│                                                                     │
│ 步骤 1：SMPL fitting                                               │
│   输入：点云或关键点                                                  │
│   输出：β (shape code), θ (pose code), 拟合误差评分                   │
│   工具：Pytorch3D + SMPLify-X / SPIN                                  │
│                                                                     │
│ 步骤 2：纹理图生成                                                   │
│   输入：多视角照片 + SMPL mesh 对齐                                   │
│   输出：UV 纹理图 (2048×2048 PNG)                                    │
│   工具：PyTorch3D texture baking                                      │
│                                                                     │
│ 步骤 3：质量校验                                                     │
│   检查：纹理清晰度、β 参数合理性、拟合误差 < 阈值                       │
│                                                                     │
│ 步骤 4：持久化                                                       │
│   - β 参数 → Avatar Database（绑定喵街会员 ID）                      │
│   - 纹理图 → CDN / R2（物件存储）                                    │
│   - 预览 GLB → CDN（按需生成）                                       │
│                                                                     │
│ 总耗时：入门级 10 秒，进阶级 2 分钟，门店级实时                        │
└─────────────────────────────────────────────────────────────────────┘
                                       ↓
┌─────────────────────────────────────────────────────────────────────┐
│ C. 前端展示（React + Three.js）                                      │
├─────────────────────────────────────────────────────────────────────┤
│ Avatar 加载链路：                                                    │
│   用户登录 → Avatar API 返回 β + 纹理 URL                            │
│   → Three.js 端 SMPL 生成器                                          │
│   → 实时构建三角网格                                                 │
│   → 加载纹理贴图                                                     │
│   → 叠加服装 GLB（CLO3D 格式，标准化骨架）                            │
│   → 驱动 pose → 360°展示 / 走秀动画 / 试衣                          │
│                                                                     │
│ 前端不下载 GLB 文件，而是用 β 参数实时重建 mesh                       │
│ 这意味着 4500 万会员不占用 4500 万 × 10MB = 45TB 存储                │
│ 只用存储：β (2KB) + 纹理图 (8MB) = 每个会员 ~8MB                     │
└─────────────────────────────────────────────────────────────────────┘
                                       ↓
┌─────────────────────────────────────────────────────────────────────┐
│ D. 换衣系统                                                         │
├─────────────────────────────────────────────────────────────────────┤
│ 服装资产：                                                          │
│   - 每件衣服 = 独立的 GLB 网格（已蒙皮到 SMPL 骨骼）                  │
│   - 银泰门店商品 → 3D 扫描 → 标准化服装资产                           │
│   - 存储格式：GLB + 元数据（型号 S/M/L, 颜色, 面料）                   │
│                                                                     │
│ 用户换衣：                                                          │
│   选择服装 → 加载 GLB → 叠加到 SMPL Avatar → 显示穿搭效果             │
│   Three.js 端做 clothing-body 碰撞检测/穿透修正                      │
│   （如果银泰接入 CLO3D API，可用物理仿真做更真实的下坠效果）            │
│                                                                     │
│ 分享：                                                              │
│   穿搭截屏 + Avatar 全景 → AI 生成时尚海报 → 分享到朋友圈              │
└─────────────────────────────────────────────────────────────────────┘

```

## 7. 为什么放弃当前方案

| 当前方案 | 根本问题 | 替代方案 |
|---------|---------|---------|
| Visual Hull + 8 views | 几何质量天花板极低 | SMPL-X fitting（先验人体知识） |
| GrabCut 轮廓 | 丢弃纹理信息，永远无肤色/细节 | 纹理烘焙（从多视角照片投影到 UV） |
| 前置摄像头 | 画质差、视角受限、姿态不可控 | 后置相机扫码/视频/门店体测仪 |
| Marching Cubes → GLB | 拓扑随机、不可换衣 | SMPL 固定拓扑、服装资产独立 |
| 独立重建 | 每次不同结果，无身份一致性 | Avatar ID + β 参数存储 |
| 8MB GLB/次 | 重复存储爆炸 | 2KB β + 纹理 URL = 8MB/会员 |

## 8. 为什么采用新方案（SMPL）

**核心答案：SMPL 把"人体"和"服装"拆成两个独立系统。**

```
旧方案：     人体 = GLB（人体和服装混在一起）
             → 换衣 = 重新生成整个 GLB
             → 不可能

新方案：     人体 = SMPL β (2KB) + 纹理图 (8MB)
            服装 = 独立 GLB (标准化骨架)
             → 换衣 = 只换服装 GLB，人体不动
             → 可做到
```

**这种分离是一切商业价值的起点：**
- 服装品牌可以制作标准化的数字服装
- 用户可以一次建模永久使用
- 换衣体验从"重新拍照"变成"点一下切换"
- 门店可以从"卖实体衣服"扩展到同时卖"数字版服装"
